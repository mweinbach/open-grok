# Open Grok v0.1.220-open-grok.29

Open Grok is Grok Build with ChatGPT Codex, Kimi, and Fireworks AI provider
support, packaged as the independent `open-grok` command with isolated state
under `OPENGROK_HOME`.

Antigravity (agy) subagent UX release: live progress, quota visibility,
and correct model/effort handling.

## Live heartbeat for Antigravity subagents

An agy subagent used to be a black box between spawn and completion — often
minutes of nothing. The runner now tails the agy language-server log while
a run is active and pushes a live phase into the subagent (and swarm-member)
card: `Starting` → `Model resolved` → `Language server ready` → `Working` →
`Wrapping up`, alongside the elapsed timer. At most one update every 2s.

## Antigravity quota in `/usage`

While an agy subagent runs, Open Grok makes one best-effort
`RetrieveUserQuotaSummary` call on the language server's local Connect port
(parsed from the run's log) and caches it. `/usage` then shows an
**Antigravity** section — per-bucket percent remaining and reset time —
in the same style as the xAI and Codex sections, with an "as of Nm ago,
refreshed when agy subagents run" note. In-process sessions only; the
section is simply absent in leader mode.

## Correct model and reasoning-effort handling

- **Effort mapping:** the fork's reasoning tiers now map cleanly to agy's
  `low|medium|high` (`max`/`ultra`/`xhigh` → `high`), and the *first* spawn
  attempt is already correct — effort-suffixed ids (`gemini-3.6-flash-high`)
  never receive `--effort`, base ids do. The suffix-swap retry stays as a
  safety net.
- **Availability detection:** a cached `GetAvailableModels` probe lets a
  spawn fail fast when the requested model is exhausted or unavailable,
  with the remaining-quota context and a suggestion to use the reference
  model — instead of a late, opaque failure.
- **Reference model:** `antigravity:gemini-3.6-flash` is now the documented
  reference — floated first in the roster, shown in the settings example,
  and used in the `agent-swarm` skill, which now spells out the
  base-vs-suffixed-id effort rule.

## OpenAI Codex account always shown in `/usage`

The `/usage` OpenAI Codex section already reported the signed-in account
(`Account: <email>`, `Plan: <plan>`) and rate-limit windows — but only when
the live usage fetch succeeded. When that fetch failed (rate limit,
transient error, offline), the whole section collapsed to "Usage
unavailable" and the account identity was lost. The signed-in account is
answerable from the stored credential with no network call, so it now
renders in the error path too: `/usage` always tells you which OpenAI
account is connected, even when the usage numbers can't load.
